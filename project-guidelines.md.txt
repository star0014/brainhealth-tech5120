## 8. Security Default Rules

Security must be considered by default in any code, architecture, or feature discussion.

Always consider these areas where relevant:

### 8.1 Authentication and Authorisation
- protect user-specific routes and data
- do not assume any user should access all data
- recommend role or ownership checks where needed
- prefer clear session/token handling
- never suggest storing passwords in plaintext

### 8.2 Input Validation
- validate all user inputs on the server side
- client-side validation is only for usability, not trust
- treat all request data as untrusted
- validate format, length, type, and allowed values

### 8.3 Database Security
- use parameterised queries / ORM-safe patterns
- prevent SQL injection
- minimise exposure of sensitive fields
- follow least privilege principles where possible
- do not hardcode credentials

### 8.4 Frontend Security
- prevent XSS with output encoding/safe rendering
- avoid unsafe HTML rendering unless clearly sanitised
- avoid storing sensitive tokens insecurely
- be careful with browser storage

### 8.5 API Security
- validate request payloads
- validate user authorisation for each protected action
- consider rate limiting for abuse-prone endpoints
- avoid verbose error leakage
- protect endpoints that expose user data or scoring logic

### 8.6 File and Media Handling
If the project supports uploads:
- validate file type
- validate size
- validate content where possible
- never trust filename alone
- avoid direct execution/storage risks

### 8.7 Secrets and Configuration
- do not hardcode secrets
- use environment variables
- do not expose secret values in logs or frontend code
- redact example secrets in responses

### 8.8 Logging and Monitoring
- prefer privacy-aware logs
- avoid logging passwords, tokens, or unnecessarily sensitive user data
- log security-relevant events when useful

### 8.9 Practical Security Balance
If security and convenience conflict, recommend balanced and demonstrable mitigations suitable for a university prototype.
Do not force enterprise-only solutions unless requested.

## 9. Web/App Security Priorities for This Project

Because this project is likely a browser-based app with user inputs and dashboards, treat the following as priority risks unless repository files show otherwise:

- SQL injection
- cross-site scripting (XSS)
- insecure authentication flows
- broken access control
- insecure direct object references
- API abuse / excessive requests
- sensitive data exposure
- weak password handling
- misconfigured deployment settings
- improper input validation
- unsafe recommendation or scoring logic if it can be manipulated
- privacy issues around lifestyle or personal data

When reviewing a feature, think through:
1. what data enters the system
2. where it is processed
3. where it is stored
4. who can access it
5. what could be abused
6. how to mitigate it simply and clearly

## 10. Threat Modelling Rules

When asked to analyse risk or design security controls, use a lightweight but structured approach.

Preferred approach:
- identify assets
- identify actors
- identify trust boundaries
- identify entry points
- identify threats
- estimate impact and likelihood
- recommend mitigations
- prioritise practical fixes

Use STRIDE when appropriate:
- Spoofing
- Tampering
- Repudiation
- Information Disclosure
- Denial of Service
- Elevation of Privilege

For each major feature, try to identify:
- what could go wrong
- how it could happen
- what the impact would be
- how the team could realistically reduce the risk

Keep the analysis assessable and explainable.
Prefer concise, strong risk reasoning over bloated theory.
